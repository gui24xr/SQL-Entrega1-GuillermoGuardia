TABLA: roles_usuario_sistema

Contiene los tipos de roles que tendra cada usuario de nuestro sistema de administracion y segun el tipo de rol sera el tipo de permisos que tiene para ingresar y operar en el sistema. Se proyecta tener user_dev, user_premium,user_admin,user_recepcion,user_medico,user_paciente,user_empleado.

Campo Tipo de dato Descripcion
id_user_rol VARCHAR(30) identificacion del tipo de usuario del sistema.
rol VARCHAR(30) Nombre del rol. Se utilice un ID ya que al rol podriamos decidir cambiarle el nombre eventualmente.

PRIMARY KEY: categoria.

TABLA: roles_empleados

Contiene los tipos de Roles de Empleados que existen en el consultorio en las distintas areas con lo cual mas adelante cruzaremos informacion y a cada empleado segun su rol vamos a darle un distinto tipo de usuario para el sistema.

Campo Tipo de dato Descripcion
id_funcion INT Identificacion del tipo de funcion.
funcion VARCHAR(30) Nombre de la funcion. Se utilice un ID ya que podriamos decidir cambiarle el nombre eventualmente a una determinada funcion.

PRIMARY KEY: id_funcion.

TABLA: prestaciones

Contiene los tipos de prestaciones de obra social y planes que estaran disponibles. De este modo podremos alimentar desde nuestro backend a nuestro frontend para que el usuario del sistema elija la prestacion con la cual se llevara a cabo la atencion por parte del medico. Si bien es cierto que podriamos tener aparte una tabla obras sociales es practico por el momento manejarlo de esta manera ya que de otra forma habria que trabajar con APIS de obras sociales y por el momento no es suficiente trabajar con este metodo. Tambien contiene esta tabla un campo llamado -activa- que es un boolean que indicara si en el momento de otorgar el turno o hacer la prestacion la misma se encuentra activa.

Campo Tipo de dato Descripcion
id_prestacion INT identificacion del tipo de prestacion.
obra_social VARCHAR(255) Nombre de obra social asociada.
plan VARCHAR(255) Plan de obra social asociado.
activa BOOLEAN Estado de la prestacion, la misma puede o no, estar activa en un momento determinado.

PRIMARY KEY: id_prestacion.

Campo Tipo de dato Descripcion
numero_consultorio INT Identificacion de consultorio
en_servicio BOOLEAN Indica si el consultorio se encuntra en uso o no, en un momento determinado. Tal vez mas adelante se podria escalar, cambiar el tipo de campo y ponerle estados como por ejemplo -en limpieza,reparacion,etc-.

PRIMARY KEY: numero_consultorio.

Cada registro de esta tabla es un registro del medico cuando atiende a un paciente y deja registro de diagnostico,observaciones,medicacion. Luego Si queremos obtener la historia clinica de un paciente determinado filtramos por su legajo y obtenemos toda su historia clinica completa.

TABLA: turnos_otorgados

En esta tabla se registraran los turnos que se otorgaran, que paciente, para que medico, posible prestacion, y estado del turno el cual ira cambiando desde que es otorgado hasta que llega, se anuncia el paciente o en su defecto si es cancelado o reprogramado.

Campo Tipo de dato Descripcion
id_turno VARCHAR(30) Identificacion del turno la cual estara dada por la combinacion del a√±o, el dia giuliano y numero de turno otorgado en el dia. Mas adelante se ejecutara un thrigger antes de la insercion para generar el ID correspondiente.
legajo_medico INT Legajo del medico que atendera el turno.
legajo_paciente INT Legajo del paciente que otorgara el turno.
fecha_hora_reservada DATETIME Fecha y hora para la cita medica. Seguramente desde el backend se maneje la insercion de este valor ya que existe la necesidad de crear un algoritmo para gestionar la ocupacion de turnos.
id_prestacion_tentativa INT Identificacion de la prestacion con la cual se llevara adelante la consulta. Es tentativa porque es posible que al momento del turno la prestacion no este activa y haya que cambiarla.
fecha_hora_otorgamiento DATETIME Fecha y hora en que se otorga el turno para tener un seguimiento del mismo.
dni_otorgante_turno VARCHAR(30) DNI de quien otorga el turno para poder cruzar datos con tabla usuarios_sistema y tener registro si el turno fue dado por un empleado en una recepcion o por un paciente a travez del sistema.
estado_turno INT Estado del turno asociado a la tabla estados_turno.
consultorio_asignado INT Consultorio en el cual se llevara a cabo la consulta. Este campo obtendra un valor cuando el paciente llega a la recepcion, se anuncia y se confirma por cual consultorio sera atendido.

PRIMARY KEY: id_turno.

FOREINGS KEYS

- Campo: legajo_medico refTabla medicos.

- Campo: legajo_paciente refTabla pacientes.

- Campo: id_prestacion_tentativa refTabla prestaciones.

- Campo: estado_turno refTabla estados_turno.

- Campo: consultorio_asignado refTabla consultorios.
